# Mixtar M1: instalacja, aktualizacja i recovery

## Status

M1 jest zakończonym wydaniem konsolowym `x86_64`. Pełny pipeline od pustego
cache przeszedł QEMU/OVMF, Hyper-V Generation 2, aktualizację A/B, odrzucenie
uszkodzonego bundle, recovery, test zewnętrznego modułu i kontrolę podpisów.
Końcowy kontrakt znajduje się w `Output/P3/M1.release.json`.

## Budowa

Wymagania hosta opisuje `BUILDER.md`. Pełne wydanie powstaje jednym poleceniem:

```powershell
.\scripts\build-m1.ps1 -CleanCache -TestHyperV
```

Wynikiem są między innymi:

- `Output/P1/MixtarRVS-M1-x86_64.disk.img` dla QEMU i fizycznego nośnika;
- `Output/P1/MixtarRVS-M1-x86_64.vhdx` dla Hyper-V Generation 2;
- `Output/P1/MixtarRVS-M1-B.update.fat` jako bundle aktualizacji;
- `Output/P1/MixtarRVS-M1-x86_64.Recovery.EFI` jako niezależne recovery.

Zapis obrazu RAW na fizyczny dysk nie jest jeszcze platformą akceptacyjną M1.
Przed takim użyciem należy traktować operację jako eksperymentalną i wykonać
kopię danych nośnika.

## Uruchomienie w Hyper-V

1. Utwórz maszynę Generation 2 z co najmniej 1 GiB RAM i dwoma vCPU.
2. Podłącz istniejący `MixtarRVS-M1-x86_64.vhdx` jako dysk startowy.
3. Podłącz maszynę do przełącznika z DHCP, na przykład `Default Switch`.
4. Wyłącz Secure Boot, dopóki firmware nie ufa certyfikatowi M1.
5. Uruchom maszynę; pierwszy boot wymaga ustawienia hasła.

Automatyczny test można uruchomić niezależnie i zachować jego VM do inspekcji:

```powershell
.\scripts\test-p2-hyperv.ps1 `
  -ManifestPath .\Output\P1\MixtarRVS-M1-x86_64.manifest.json `
  -KeepVM
```

Test korzysta z COM1 i działa headless. Hyper-V Manager pokazuje stan maszyny,
ale okno VMConnect nie jest źródłem logu akceptacyjnego. Bez `-KeepVM` maszyna
jest usuwana po teście.

## Uruchomienie w QEMU/OVMF

Obraz RAW jest samodzielnym dyskiem UEFI. Próba akceptacyjna uruchamia go przez
OVMF bez przekazywania `-kernel` ani zewnętrznego initramfs. Pełny build
wykonuje tę próbę automatycznie i zapisuje wynik w `Output/P1/Qemu-p2.json`.

## Pierwszy boot i konta

Pierwszy boot komunikuje się po angielsku i wymaga ustawienia hasła. Nazwy
`root`, `Administrator` i `Superuser` prowadzą do tej samej tożsamości UID 0 i
jednego hasła. Login uruchamia zsh z konfiguracją GRML; ash pozostaje
technicznym POSIX `sh` dla initu i skryptów.

## Model danych

- `mixtar/ROOT/M1-A` i `mixtar/ROOT/M1-B` są readonly slotami systemu.
- `/Users`, `/System/State` i `/System/Logs` pozostają trwałe między slotami.
- `/System/Cache` jest odbudowywalne i oddzielone od niezmiennego systemu.
- EFI Stub slotu wskazuje odpowiadający mu dataset root.
- Kernel, OpenZFS, initramfs i userspace ZFS przełączają się jako jeden zestaw.

## Aktualizacja

Podpisany wolumin `MixtarRVS-M1-B.update.fat` ma label `MIXTARUPD` i zawiera
`Update.config`, `Update.sig`, `root.tar` oraz `BOOTX64.EFI`. Po podłączeniu
woluminu zamontuj go w systemie, na przykład jako `/Volumes/Update`, i uruchom:

```text
/System/Core/Update/mixtar-update /Volumes/Update
```

Updater wykonuje kolejno:

1. Weryfikację podpisu manifestu oraz SHA-256 rootfs i EFI.
2. Utworzenie i wypełnienie niezmiennego datasetu kandydata.
3. Zapis transakcji `pending` w `/System/State/Update/Transaction.config`.
4. Zachowanie bieżącego EFI jako `Previous.EFI`.
5. Weryfikację pliku `.new`, atomowe zastąpienie `BOOTX64.EFI` i zmianę `bootfs`.
6. Akceptację slotu po poprawnym starcie jego usług platformowych.

Błąd przed końcowym przełączeniem nie zmienia aktywnego slotu. Uszkodzony
bundle jest odrzucany przed utworzeniem aktywnej aktualizacji. Po zainstalowaniu
poprawnego bundle wykonaj reboot, aby uruchomić kandydata.

## Recovery i rollback

Recovery znajduje się na ESP jako `\EFI\Mixtar\Recovery.EFI`; w uruchomionym
Mixtarze odpowiada mu `/System/EFI/Mixtar/Recovery.EFI`. Można uruchomić je z
menu firmware lub powłoki UEFI. Konsola recovery oferuje:

```text
rollback
shell
```

`rollback` przywraca `Previous.EFI`, ustawia poprzedni dataset jako `bootfs` i
nie modyfikuje `/Users` ani trwałego stanu niezależnego od slotu. Automatyczna
ścieżka testowa używa parametru:

```text
mixtar.recovery=rollback
```

Recovery ma własny EFI Stub i initramfs; nie wymaga działającego userspace
aktywnego slotu. `Output/P3/Qemu-p3.json` potwierdza start slotu B, jego
akceptację, rollback i ponowny start slotu A.

## Podpisy i Secure Boot

Pipeline podpisuje moduły, EFI `M1-A`, EFI `M1-B`, recovery, manifest obrazu i
końcowy raport wydania. `Output/P3/Release-validation.json` potwierdza podpisy
EFI i modułów, integralność artefaktów, readonly root oraz brak publicznego FHS.

Automatyczna akceptacja Hyper-V ma Secure Boot wyłączony. Pliki są przygotowane
do Secure Boot, ale egzekwowanie zaufania wymaga enrollmentu certyfikatu
`Output/P3/Signing/M1.certificate.cer` w konkretnym firmware. Standardowy
szablon Hyper-V nie ufa lokalnemu certyfikatowi M1.

Domyślny pipeline tworzy lokalną tożsamość M1 przy pierwszym buildzie. Publiczne
wydanie powinno otrzymać chroniony klucz organizacji i kopię offline. Prywatnego
`M1.key.pem` nie wolno publikować ani umieszczać w obrazie.

## Raporty akceptacji

- `Output/P1/Qemu-p2.json`: dwa booty, login, sieć, trwałość i poweroff.
- `Output/P1/HyperV-p2.json`: dwa booty tego samego VHDX w Generation 2.
- `Output/P3/Qemu-p3.json`: uszkodzony update, slot B i rollback do A.
- `Output/P3/Module-SDK.json`: zewnętrzny, podpisany moduł kernela.
- `Output/P3/Release-validation.json`: hashe, FHS, host paths i podpisy.
- `Output/P3/M1.release.json`: końcowy kontrakt pełnego przebiegu.
