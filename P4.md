# P4: kontrakt warstwy produktu Mixtar

Status: normatywny kontrakt P4-pre  
Data: 17 lipca 2026

`ROADMAP.md` określa kolejność prac. Ten dokument określa zachowanie produktu. Pliki z `MixtarRVS.zip`, `kontekst.md`, prototypy i wygenerowane artefakty nie są źródłami normatywnymi.

## 1. Cel i granica

P4 dodaje warstwę produktu do ukończonej dystrybucji konsolowej P3. Nie przebudowuje bootowania, OpenZFS, OpenRC ani konsoli i nie może odbierać możliwości uruchomienia Mixtara bez grafiki.

Mixtar jest własnym systemem operacyjnym używającym kernela Linux jako prywatnej warstwy sprzętowej. Publiczne API, namespace, aplikacje, konfiguracja i komunikaty należą do Mixtara. POSIX, FHS i nazwy dystrybucji użytej do budowania nie są publiczną tożsamością produktu.

Wymagania zapisane jako `MUSI` są obowiązkowe dla P4. `POWINIEN` oznacza zachowanie domyślne, od którego odstępstwo wymaga uzasadnienia i pomiaru.

## 3. Stabilny publiczny namespace

Publiczny root Mixtara MUSI zawierać tylko główne przestrzenie:

```text
/Applications
/System
/Users
/Volumes
/Temporary
```

Znaczenie przestrzeni:

| Ścieżka | Kontrakt |
| --- | --- |
| `/Applications` | Zainstalowane aplikacje użytkownika w formie `*.apx` |
| `/System` | Niezależna od użytkownika baza systemu, runtime, konfiguracja i UI |
| `/Users` | Konta oraz ich trwałe dane |
| `/Volumes` | Zamontowane pule, datasety i nośniki |
| `/Temporary` | Dane nietrwałe bez gwarancji zachowania po restarcie |

Wewnętrzne nazwy Mixtara są określone przez `Root/System/Configuration/Layout.config`. P4 MUSI używać między innymi:

```text
/System/Commands
/System/Configuration
/System/Libraries
/System/Runtime
/System/Processes
/System/Hardware
/System/Devices
/System/State
/System/Cache
/System/Logs
/System/Terminal
```

`/System/Runtime`, `/System/Processes`, `/System/Hardware` i `/System/Devices` są właściwymi punktami runtime Mixtara. Host nie publikuje aliasów `/run`, `/proc`, `/sys` ani `/dev`. Prywatny provider zgodności może utworzyć te nazwy we własnej przestrzeni montowań wyłącznie dla obcego drzewa procesów.

Nie wolno tworzyć w publicznym root hardlinków ani symlinków udających FHS. Natywne komponenty odwołują się bezpośrednio do nazw Mixtara.

## 4. Kontrakt plików `.config`

Każdy systemowy plik `.config` P4 MUSI być tekstem UTF-8 o składni TOML. Rozszerzenie `.config` nie oznacza SQLite ani innej bazy danych.

Reguły:

- `schema` jest dodatnią wersją schematu danego kontraktu.
- Parser MUSI odrzucać błędny TOML, zduplikowane klucze i nieobsługiwaną wersję schematu.
- Pola bezpieczeństwa, entrypointy i capabilities MUSZĄ być walidowane fail-closed.
- Zapis trwałej konfiguracji MUSI być atomowy.
- Czytnik może skompilować konfigurację do cache pod `/System/Cache`, ale cache jest odtwarzalny i powiązany z hashem źródła.
- Tekstowy `.config` zawsze pozostaje źródłem prawdy.
- SQLite może być użyte przez aplikację do jej danych, ale nie jest formatem kontraktu Mixtara.

Wersje komponentów są przypinane w blokadzie konkretnego wydania Mixtara. Nie są wpisywane do kontraktu architektury jako wiecznie aktualne numery.

## 5. APX v1

APX jest natywnym obiektem aplikacji Mixtara. Fizycznie jest zwykłym katalogiem, natomiast Files, Executor i UI prezentują go jako jeden uruchamialny obiekt.

Minimalny kształt:

```text
Example.apx/
  Example.config
  Program/
    Example
  Icon/
    app.svg
  Resources/
  Data/
```

`Example.config` MUSI odpowiadać nazwie katalogu. `Data` zawiera wyłącznie dane dostarczone z aplikacją; stan zapisywalny trafia do przestrzeni użytkownika przekazanej przez Executor.

Minimalna konfiguracja:

```toml
schema = 1

[application]
id = "org.mixtar.Example"
name = "Example"
version = "1.0"

[entry.graphical]
kind = "native"
path = "Program/Example"

[entry.terminal]
kind = "native"
path = "Program/Example"

[permissions]
required = ["ui.window"]
optional = []
```

Reguły APX v1:

- `application.id`, `application.name` i `application.version` są wymagane.
- `entry.graphical` albo `entry.terminal` MUSI istnieć; mogą wskazywać ten sam program.
- P4 v1 rozpoznaje tylko `type = "native"`.
- Natywny program Mixtara na x86-64 jest ELF-em zgodnym z runtime Mixtara; aplikacja Avalonia jest publikowana jako Native AOT.
- Ścieżka entrypointu jest względna wobec bundla.
- Ścieżka absolutna, pusta, zawierająca `..`, wychodząca poza bundle albo prowadząca przez niedozwolony symlink MUSI zostać odrzucona.
- Nieznany rodzaj (`kind`) entrypointu, capability lub wymagany feature MUSI zostać odrzucony.
- Kontekst uruchomienia MUSI przekazać `HOME` wskazujące katalog użytkownika w layoucie Mixtara oraz `TMPDIR=/Temporary`; Executor i aplikacje nie mogą wymagać `/etc/passwd` do ustalenia tych ścieżek.
- Identyfikatory uruchomień i plików atomowych MUSZĄ pobierać entropię z `/System/Devices/urandom`; implementacja nie może zakładać istnienia `/dev/urandom`.
- Executor otwiera konfigurację i zasoby tylko do odczytu.
- Executor nigdy nie składa polecenia powłoki; przekazuje program i argumenty jako wektor.
- APX nie zawiera wariantów Windows, Linux, FreeBSD i macOS w jednym bundlu.
- Usunięcie katalogu nie zastępuje transakcji instalacyjnej, jeśli aplikacja ma aktywne procesy lub politykę systemową.

## 6. Uruchomienie graficzne i terminalowe

Dwuklik i terminal uruchamiają ten sam APX przez ten sam Executor. Różni je jawny kontekst uruchomienia, a nie osobny format aplikacji.

Konteksty:

| Kontekst | Zachowanie |
| --- | --- |
| `graphical` | Executor wybiera `entry.graphical`, przekazuje dostęp do Wayland i nie otwiera terminala |
| `terminal` | Executor wybiera `entry.terminal`, zachowuje argumenty, stdin, stdout, stderr i kod wyjścia |

Jeżeli bundle ma tylko jeden entrypoint, Executor może użyć go w obu kontekstach i MUSI przekazać kontekst aplikacji.

Deskryptor uruchomienia MUSI zawierać co najmniej:

```text
launch.id
launch.context
application.id
bundle.path
session.id
user.name
app.data.path
diagnostics.enabled
```

Aplikacja uruchomiona z terminala POWINNA obsługiwać `--help` oraz `/?` jako równoważne żądania pomocy. Aplikacja GUI POWINNA otworzyć UI bez wymagania argumentów.

Tryb diagnostyczny jest jawną opcją Executora. Przekazuje logi i metadane lifecycle do terminala, ale nie przyznaje dodatkowych capabilities, nie omija polityki i nie podnosi uprawnień.

## 7. Executor v1

Jedynym właścicielem semantyki APX jest:

```text
/System/Runtime/Executor
```

Executor MUSI wykonać kolejno:

1. Ustalić kanoniczną ścieżkę bundla.
2. Otworzyć pasujący `.config` jako TOML.
3. Zweryfikować schemat, tożsamość, entrypoint i integralność bundla.
4. Odczytać politykę capabilities dla bieżącego użytkownika i sesji.
5. Utworzyć identyfikator oraz deskryptor uruchomienia pod `/System/Runtime`.
6. Uruchomić entrypoint bez pośrednictwa shella.
7. Przekazać proces do właściciela lifecycle bieżącej sesji.
8. Zapisać ustrukturyzowany wynik bez ujawniania sekretów.

Uruchomienie terminalowe domyślnie czeka i zwraca kod aplikacji. Uruchomienie graficzne zwraca identyfikator launch po skutecznym przekazaniu procesu do sesji; opcja `wait` może czekać na zakończenie.

Executor nie instaluje aplikacji, nie uwierzytelnia użytkownika, nie jest kompozytorem i nie interpretuje UI.

## 8. Sesja v1

Sesja jest ograniczonym zbiorem procesów jednego uwierzytelnionego użytkownika. Stan aktywnej sesji jest nietrwały i mieści się pod:

```text
/System/Runtime/Sessions/<session-id>
```

Kontrakt sesji MUSI przechowywać:

```text
session.id
session.type = console | graphical
session.user
session.created_at
session.state = starting | active | stopping | stopped | failed
session.compositor
session.desktop
```

Łańcuch graficzny jest jeden:

```text
OpenRC -> MDDM -> MWM/Wayland -> Login UI/Avalonia -> auth provider -> session manager -> desktop/Avalonia -> APX
```

MMWM jest kompozytorem Wayland i właścicielem powierzchni, fokusu, położenia, dekoracji oraz operacji okien. MDDM jest modelem sterowników obrazu pomiędzy Linux DRM/KMS a MWM; nie jest procesem logowania ani UI. Osobny Login UI zbiera dane logowania i przekazuje je providerowi uwierzytelniania. Login UI nie przechowuje haseł ani nie implementuje kryptografii haseł.

Wylogowanie zatrzymuje procesy należące do sesji, czyści jej runtime i wraca do Login UI. Nie zatrzymuje usług systemowych ani konsoli. Awaria Avalonii, Login UI, MWM albo MDDM nie może uniemożliwić bootu do niezależnej konsoli P3.

## 9. Tożsamość administracyjna

Kanoniczną publiczną nazwą uprzywilejowanego konta Mixtara jest `Administrator`.

```text
Administrator
Superuser
root
```

Wszystkie trzy nazwy rozwiązują się do tej samej tożsamości uprzywilejowanej; nie tworzą trzech kont. UI wyświetla `Administrator`. Logowanie i narzędzia administracyjne akceptują aliasy, a audyt zapisuje nazwę kanoniczną oraz alias użyty przy wejściu. `root` pozostaje również nazwą zgodności dla prywatnych wysp FHS.

Komunikaty systemu, logowania, Executora i aplikacji bazowych MUSZĄ być po angielsku. Lokalizacje są warstwą późniejszą, nie zmianą kontraktu.

## 10. Capabilities i uprawnienia

APX deklaruje potrzebę operacji, ale sam nie nadaje sobie uprawnień. Domyślnie aplikacja otrzymuje odczyt własnego bundla, zapis własnych danych użytkownika i kontrolę własnych procesów.

Początkowy namespace capabilities:

```text
ui.window
ui.notifications
network.client
network.listen
storage.user
storage.volumes.read
storage.volumes.write
devices.input
devices.graphics
process.inspect
system.status
system.admin
```

Reguły:

- Brak wymaganej capability przerywa uruchomienie z angielskim komunikatem.
- Brak opcjonalnej capability uruchamia aplikację w trybie ograniczonym.
- Nieznana capability jest błędem kontraktu.
- Grant jest związany z `application.id` i tożsamością wydawcy, nie ze ścieżką pliku.
- Nowa wymagana capability po aktualizacji wymaga ponownej decyzji.
- `system.admin` nie uruchamia całej aplikacji jako Administrator; uprzywilejowaną operację wykonuje wąski broker po sprawdzeniu polityki.
- Tryb diagnostyczny i uruchomienie z terminala nie zmieniają uprawnień.
- Hasła, tokeny i klucze prywatne nie mogą znajdować się w APX ani śledzonych `.config`.

## 11. Stos graficzny

Jedynym stosem produktu P4 jest:

```text
DRM/KMS -> MWM/Wayland -> Avalonia 12.1 -> aplikacja Native AOT na stabilnym .NET 10
```

Avalonia jest frameworkiem klientów UI. Nie zastępuje MWM. Konkretne poprawki Avalonii i .NET są przypinane w blokadzie wydania Mixtara. Wydania preview .NET nie wchodzą do Mixtar Base.

Xwayland może działać wyłącznie w prywatnym providerze zgodności. Natywne Login UI, pulpit i aplikacje systemowe nie mogą wymagać X11, Qt, QML, GTK, przeglądarki ani publicznej instalacji współdzielonego runtime .NET. MDDM i MWM są warstwami natywnymi poniżej Avalonii.

Zależności Native AOT, Wayland, EGL/GBM, fontconfig, FreeType i HarfBuzz MUSZĄ zostać zbudowane dla `/System` z kontrolowanym interpreterem ELF i zależnościami. Nie wolno rozwiązywać tego trwałymi aliasami FHS w hoście.

## 12. Obce programy

APX v1 opisuje natywne aplikacje Mixtara. ELF wymagający FHS, Windows PE/MZ i programy macOS nie są uruchamiane przez udawanie natywnego APX.

Obsługa obcego ABI należy do jawnego providera zgodności, który tworzy prywatny runtime i prywatny namespace procesu. Provider nie może publikować FHS w hoście, zmieniać kontraktu APX ani osłabiać polityki capabilities. Windows i macOS pozostają zakresem późniejszym niż podstawowy P4.

## 13. Stabilność kontraktu

- `P4.md` jest jedynym normatywnym kontraktem produktu P4-pre.
- `ROADMAP.md` śledzi wykonanie, ale nie redefiniuje semantyki.
- Zmiana niekompatybilna wymaga nowej wersji schematu i jawnej migracji.
- Rozszerzenia kompatybilne są addytywne i nie mogą zmieniać znaczenia istniejących pól.
- Prototyp nie staje się API tylko dlatego, że działa.
- Wygenerowany cache, benchmark i artefakt `out` nie są źródłem kontraktu.

## 14. Kolejność implementacji po P4-pre

1. Ustalić liczbowe budżety startu, RAM, CPU, I/O i rozmiaru runtime.
2. Zaimplementować parser i walidator APX v1 bez uruchamiania GUI.
3. Zaimplementować Executor v1 dla konsoli i zachować P3 jako ścieżkę awaryjną.
4. Zaimplementować sesję, capabilities i testową aplikację Native AOT.
5. Zbudować MWM/Wayland oraz zależności Avalonii pod `/System`.
6. Dopiero wtedy implementować MDDM, MWM, osobny Login UI i pulpit.

P4-pre nie jest zamknięte, dopóki budżety nie zostaną zapisane, a niezależność konsoli nie zostanie pokryta bramką regresji.

## 15. Budżety wydajności P4

Maszynowym źródłem budżetów jest `Root/System/Configuration/Product/P4.config`. Każda metryka ma cel optymalizacyjny i twardy limit wydania. Przekroczenie celu wymaga analizy, natomiast przekroczenie twardego limitu blokuje wydanie.

Pomiary prototypu Avalonia 12.1 na stabilnym .NET 10 Native AOT są zapisane wyłącznie jako pochodzenie wartości. Nie zaliczają bramki Mixtara. Bramkę zaliczają dopiero pomiary wydania uruchomionego na referencyjnym profilu QEMU i Hyper-V.

Obowiązujące grupy pomiarów:

| Grupa | Najważniejszy twardy limit |
| --- | --- |
| Boot do konsoli | 10 s od EFI do gotowego logowania |
| Boot do Login UI | 12 s od EFI do gotowego interfejsu logowania |
| Login | 3 s od akceptacji tożsamości do interaktywnego pulpitu |
| Referencyjny APX | 500 ms warm, 1500 ms cold |
| Referencyjny APX | 128 MiB working set, 160 MiB private |
| Bezczynna sesja graficzna | 384 MiB i 2% jednego logicznego rdzenia |
| Płynność | p95 cel 16,7 ms, p99 limit 33,3 ms |
| Stabilność APX | maksymalnie 8 MiB zatrzymanego wzrostu po 100 cyklach |
| Bezczynne I/O | maksymalnie 4 MiB zapisu na godzinę |

Sprzętowa adaptacja może wybrać mniejsze cache, lżejszy renderer albo wyłączyć funkcję opcjonalną. Nie może po cichu podnieść limitu. Zmiana limitu wymaga zmiany kontraktu, uzasadnienia pomiarem i wpisu wydania.

## 16. Niezależność konsoli

`Profiles/qemu-x86_64.toml` pozostaje profilem konsolowego M1. Graficzny P4 otrzyma osobny profil i osobny overlay `Product/Root`; payload MDDM, MWM, Wayland, Login UI, Avalonia i .NET nie może trafić do bazowego `Root` obrazu konsolowego.

`scripts/validate-console-independence.py` sprawdza gotowy root archive, profil, budżet oraz raport dwóch bootów QEMU. Bramka wymaga obecności initu, BusyBox, OpenRC i zsh, odrzuca graficzny payload oraz odwołania do grafiki z usług OpenRC. `scripts/build-m1.ps1` uruchamia tę bramkę i zapisuje `Output/P4/Console-independence.json` w końcowym raporcie wydania.

Pominięcie QEMU może utworzyć artefakt developerski ze statycznym wynikiem, ale nie stanowi pełnego dowodu runtime i pozostawia `qemu_p2 = false` w kontrakcie wydania.

## 17. Prototyp P4-pre

`mixtar_builder.apx` jest wykonywalnym prototypem kontraktu, a nie finalnym runtime. Waliduje katalog APX, TOML, identyfikator aplikacji, konteksty, capabilities, bezpieczne ścieżki bez symlinków oraz ELF64 x86-64. Polecenie `plan` tworzy deskryptor i argv bez uruchamiania procesu i bez shella.

`Prototypes/APX/ContractProbe.apx` jest referencyjnym bundlerem z minimalnym ELF-em wykonującym wyłącznie `exit(0)`. Prototyp dowodzi formatu i granicy Executora; nie jest kopiowany do M1 i nie jest publicznym API `/System/Runtime/Executor`.

## 18. Granica MDDM, MWM i Login UI

`MDDM` oznacza **MixtarRVS Display Driver Model**. Jest stabilną warstwą Mixtara nad sterownikami GPU kernela Linux, DRM/KMS, buforami i synchronizacją obrazu. Nie jest display managerem, ekranem logowania, kompozytorem ani aplikacją Avalonia.

MDDM odpowiada za:

- wykrywanie adapterów, wyjść obrazu, trybów, hotplug i właściwości sprzętu;
- kontrakt przydzielania, importu i eksportu buforów obrazu;
- synchronizację, fences, kolejki prezentacji i raportowanie zakończenia klatki;
- stabilne mapowanie możliwości Linux DRM/KMS na nazwy i błędy Mixtara;
- reset, utratę urządzenia, odzyskanie sterownika i diagnostykę bez uzależniania klientów od ścieżek FHS;
- udostępnienie MWM tylko tych funkcji, które są rzeczywiście obsługiwane przez aktywny sterownik.

`MWM` oznacza **MixtarRVS Window Manager**. Jest kompozytorem Wayland korzystającym z MDDM i odpowiada za powierzchnie, focus, położenie, rozmiar, dekoracje, input routing, workspaces oraz lifecycle okien. MWM nie uwierzytelnia użytkowników i nie jest modelem sterowników.

`Login UI` jest osobnym klientem Avalonia uruchamianym przez Session Manager. Korzysta z MWM jak każda uprzywilejowana powierzchnia systemowa, a dane uwierzytelniające przekazuje do oddzielnego Auth Providera. Nazwa tego komponentu nie może używać skrótu MDDM.

Normatywny łańcuch grafiki:

```text
Linux GPU drivers / DRM / KMS
  -> MDDM
  -> MWM / Wayland
  -> Login UI lub pulpit Avalonia
  -> aplikacje APX
```
